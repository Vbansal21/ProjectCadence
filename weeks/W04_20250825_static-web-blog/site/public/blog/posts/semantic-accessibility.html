Why semantic HTML exists

Browsers, screen readers, and search engines parse structure first. When the structure is meaningful—<nav> instead of a generic <div>—you get built-in accessibility, better indexing, and fewer scripts to fake behavior. Semantics reduce the amount of ARIA you need and make your pages resilient when CSS or JS fail.

Landmarks before ARIA

Prefer native elements. Only add ARIA when you can’t express meaning natively, and never override native semantics.

<header>...</header>
<nav aria-label="Main">...</nav>
<main id="main">...</main>
<aside aria-label="Related">...</aside>
<footer>...</footer>

Pitfall: adding role="button" to anchors that navigate. If it goes somewhere, it’s a link. If it triggers an action, it’s a button.

Headings as an outline

Headings communicate hierarchy to assistive tech and crawlers. Keep levels consistent. Don’t jump from h1 to h4 without need.

<h1>Title</h1>
<h2>Section</h2>
<h3>Subsection</h3>

Pitfall: styling a <div> to look like a heading. Screen readers won’t treat it as one.

Links vs buttons

  • Link: navigates to a URL (<a href>). Supports open-in-new-tab, visited state, etc.
  • Button: performs an action on the current page (<button>). Supports keyboard activation.
<a href="/docs">Read docs</a>
<button type="button" aria-pressed="false">Toggle theme</button>

Pitfall: span with click handlers. It’s neither link nor button, and isn’t keyboard-reachable by default.

Images, alt, and figures

Write purposeful alt text. If an image is decorative, use empty alt (alt="") so it’s skipped. For diagrams, pair with <figure> + <figcaption>. 

<figure>
  <img src="../../assets/media/og-image.svg" alt="Site banner" />
  <figcaption>Banner for ProjectCadence blog</figcaption>
</figure>

Forms people can use

  • Associate labels with controls (<label for> / id).
  • Use fieldset + legend for grouped options.
  • Announce errors with aria-describedby and focus the first error.
<label for="email">Email</label>
<input id="email" name="email" type="email" aria-describedby="email-hint" />
<p id="email-hint" class="text-neutral-400">We’ll never share your email.</p>

Tables with structure

Not for layout. Add <caption>. Use <th scope="col|row"> and <thead>/<tbody>.

<table>
  <caption>Color contrast ratios</caption>
  <thead><tr><th scope="col">Text</th><th scope="col">Background</th><th scope="col">Ratio</th></tr></thead>
  <tbody><tr><th scope="row">#e5e5e5</th><td>#0a0a0a</td><td>13.5:1</td></tr></tbody>
</table>

Metadata that search understands

  • Title is specific and under ~60 chars.
  • Meta description summarizes in ~155 chars.
  • Canonical URL avoids duplicates.
  • Open Graph/Twitter for rich sharing.
  • JSON-LD describes the article to crawlers.

One-page checklist

  • <html lang="en"> and a specific, accurate <title>.
  • Landmarks in place: <header>, <nav>, <main>, <footer>.
  • Skip link and visible focus rings.
  • Heading outline is consistent.
  • Links navigate; buttons act. No clickable div/span.
  • Images have purposeful alt; decorative images use alt="".
  • Forms: labels, helpful hints, error association.
  • Tables: caption, scopes, proper sections.
  • Metadata: description, canonical, OG/Twitter, JSON-LD.
